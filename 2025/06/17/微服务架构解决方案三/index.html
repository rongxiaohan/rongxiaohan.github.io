Nginx访问日志收集分析

一、Nginx 访问日志基础

1. 日志格式配置

在 nginx.conf 中配置日志格式：

2. 日志内容说明

  • $remote_addr: 客户端 IP

  • $request: 请求方法和 URI

  • $status: HTTP 状态码

  • $request_time: 请求处理总时间

  • $upstream_response_time: 后端响应时间

  • $http_user_agent: 用户代理

  • $http_x_forwarded_for: 真实客户端 IP

二、日志收集方式

1. 手动查看

2. 日志轮转（防止日志文件过大）

3. 采集工具

  • Filebeat：轻量级日志采集器，配合 ELK 使用。

  • Fluent Bit / Fluentd：日志收集与转发。

  • rsyslog：系统级日志收集。

  • Logstash：可处理更复杂的日志清洗逻辑。

三、日志存储方案

常见方案

  • 本地存储 + grep/awk：适用于小规模。

  • ELK Stack（Elasticsearch + Logstash + Kibana）：大规模分析。

  • Grafana Loki + Promtail + Grafana：轻量日志可视化。

  • 云服务日志平台（如阿里云日志服务、腾讯 CLS、AWS CloudWatch）

四、日志分析目标与指标

1. 流量分析

  • 每分钟请求数（QPS）

  • 每 IP 请求量排行

  • URI 请求频率分布

2. 性能分析

  • 请求延迟（$request_time）

  • 后端响应慢的接口（$upstream_response_time）

  • 访问状态码分布（如 5xx 错误分析）

3. 安全分析

  • 攻击行为识别（如 SQL 注入、CC 攻击）

  • 异常 IP（频繁请求 404、登录接口等）

  • UA 异常（扫描工具标识）

五、日志分析示例

1. 统计状态码数量（awk）

2. 查看访问量最多的 URI

3. 找出访问最频繁的 IP

4. 过滤响应时间大于 2s 的请求

六、ELK Stack 简要部署流程

  1. Filebeat 读取 /var/log/nginx/access.log

  2. Logstash 对日志进行结构化解析（使用 grok 规则）

  3. Elasticsearch 存储结构化日志

  4. Kibana 图形化展示，如：

    • HTTP 状态码趋势

    • URI 热门排行

    • 地理位置来源

    • 响应时间分布

接入SkyWalking实现微服务链路追踪

一、SkyWalking 简介

SkyWalking 是一个开源的 APM（应用性能监控）工具，支持分布式链路追踪、服务拓扑分析、性能指标采集、告警等能力。

支持的语言和框架包括：

  • Java（Spring Boot、Spring Cloud、Dubbo、Tomcat 等）

  • Golang、Node.js、Python（社区插件）

  • 支持通过 Envoy/Service Mesh 接入

  • 支持 OpenTelemetry 数据上报

  • 📊 拓扑图（Topology）

  • 🧩 服务实例健康状态

  • 🔔 告警规则配置

链路追踪详情包括：

  • 方法耗时

  • 各阶段调用时间

  • 调用关系（RPC / HTTP / MQ）

  • SQL 查询明细（如果配置了 JDBC 插件）

一、ELK 实现链路追踪的基本原理

核心思想：在日志中注入 TraceId/SpanId，并在整个服务调用链中透传，最后通过 Kibana 可视化还原调用路径。

步骤 1：生成并透传 TraceId

✅ 推荐方式：Spring Cloud Sleuth（如果用的是 Spring Cloud）

Sleuth 会自动：

  • 为每个请求生成 TraceId、SpanId

  • 透传到下游服务（基于 HTTP Header）

  • 自动在日志中追加 TraceId、SpanId（配合 logback）

  • 展示多个服务中与该 TraceId 相关的日志

  • 还原调用链流程

  • 制作「调用链查询面板」支持 traceId 快速查询

四、进阶方案：ELK + Zipkin / OpenTelemetry

1. Zipkin 数据写入 Elasticsearch

如果你使用 Spring Cloud Sleuth + Zipkin，可将 Zipkin 数据直接写入 Elasticsearch：

然后你可以在 Kibana 中搜索 zipkin-* 索引查看链路数据。

2. 使用 OpenTelemetry 采集 → Logstash 转发至 ES

OpenTelemetry 可将 trace 信息导出为 JSON 格式，通过 Logstash 或直接推送到 Elasticsearch，可实现更细粒度的分析。

五、总结对比

实现方式 特点说明
ELK + 日志埋点 实现链路追踪依赖 TraceId 贯穿和日志分析
ELK + Sleuth + Zipkin 利用 Sleuth 自动追踪，Zipkin 提供链路结构，ES 存储
ELK + OpenTelemetry 标准化追踪采集，需额外组件转化/导入
SkyWalking / Jaeger 原生支持链路追踪，体验更好但部署稍复杂
