一、JDK 17 新特性总览

特性编号 特性名 状态
JEP 356 增强的伪随机数生成器 正式
JEP 382 新的 macOS 渲染管线（Metal） 正式
JEP 391 macOS/AArch64 支持 正式
JEP 409 密封类（Sealed Classes） 正式（JDK15预览 → 正式）
JEP 406 Switch 模式匹配（预览） 🧪 预览
JEP 403 强封装 JDK 内部 API 正式
JEP 411 弃用安全管理器（Security Manager） 正式弃用
JEP 412 外部函数和内存 API（预览） 🧪 预览
JEP 414 Vector API（第二次预览） 🧪 预览

🎯 二、重点特性详解

🔒 1. JEP 409：密封类（Sealed Classes）

用于限制哪些类可以继承某个类或实现某个接口。

示例：

  • sealed 表示密封类

  • permits 指定哪些类可以继承

  • 子类必须标记为：

    • final（不能再被继承）

    • sealed（继续密封）

    • non-sealed（打破密封）

🔀 2. JEP 406：Switch 模式匹配（预览）

让 switch 支持模式匹配（pattern matching）

示例：

  • 可以在 switch 中判断对象类型并绑定变量

  • 比 instanceof + 强制转换 更优雅

🎲 3. JEP 356：增强的随机数 API

提供了一个新的统一的伪随机数框架，支持多种算法。

示例：

  • 新接口：RandomGenerator

  • 支持多种可插拔算法：Xoroshiro、L64X256 等

  • 更适合需要高质量随机数的场景（如加密、仿真）

🧬 4. JEP 403：强封装内部 API（sun.misc 彻底封锁）

JDK 内部包（如 sun.misc.Unsafe）不再默认可访问

  • 以前很多框架通过反射访问 sun.misc.Unsafe

  • JDK 17 开始彻底封装这些包

  • 反射访问会抛出异常，必须使用显式命令开启访问：

🧹 5. JEP 411：弃用 SecurityManager

  • SecurityManager 已使用近 20 年，但现代 Java 更倾向容器和操作系统安全

  • JDK 17 开始标记为过时，未来将被移除

  • 如果你有 Applet / 沙箱场景 → 需考虑迁移

📱 6. JEP 391：macOS AArch64 支持（M1 芯片原生支持）

  • Apple Silicon（M1/M2）架构原生支持，无需 x86 翻译

  • 提升 macOS 下的运行效率和兼容性

🧪 三、其他预览/孵化特性（关注即可）

特性 说明
JEP 412 外部函数和内存 API（预览） 类似 JNI 的替代，更安全
JEP 414 Vector API（第二次预览） 支持 SIMD 指令，提升数值计算性能
JEP 382 Metal 渲染管线 替代 macOS 上不再维护的 OpenGL，JavaFX 相关

📌 四、JDK 17 废弃/删除的内容

变更 描述
SecurityManager 弃用 安全策略基于 OS/容器更推荐
Applet 被标记为 deprecated 彻底退出历史舞台
sun.misc.* 等内部包强封装 需反射暴露参数访问

五、适合升级的理由

  • 🛡 LTS 长期支持（Oracle 支持至 2029，阿里 OpenJDK 亦长期维护）

  • 🚀 更快的 GC（G1 和 ZGC 大幅优化）

  • 🔒 更安全（封装内部 API）

  • 开发体验更好（密封类、switch 模式匹配、增强随机数）

🧰 六、企业应用升级建议

目标 升级建议
Spring Boot 应用 推荐 Spring Boot 2.6+ 或 3.x 搭配 JDK 17
Dubbo/微服务 关注内部反射访问的兼容性问题
中间件开发 避免使用 Unsafe，改用 VarHandle 或 JEP 412
