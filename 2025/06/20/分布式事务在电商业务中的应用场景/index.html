🎯 Seata 的核心解决方案概览

Seata 支持的四种事务模式对比：

模式 特点 适用场景 原理简介
AT（Automatic Transaction） 无侵入，适合关系型数据库 一般业务系统（订单、库存） 基于 本地事务 + Undo Log
TCC（Try-Confirm-Cancel） 强一致性，适合可预留资源的场景 金融扣款、库存冻结 显式定义三段接口
Saga 长事务，弱一致性 营销、审批等流程型系统 依赖状态机 + 补偿操作
XA 标准两阶段协议 需要强一致性且数据库支持 XA 的场景 基于 JDBC XA 协议

🔧 Seata AT 模式：核心机制（最常用）

适用于 Spring Boot + MySQL 分布式系统，无业务侵入。

关键组成：

  • TM（Transaction Manager）：发起全局事务（如订单服务）

  • RM（Resource Manager）：管理分支事务资源（如库存、账户服务）

  • TC（Transaction Coordinator）：事务协调器（中心）

执行流程图：

Undo Log 机制：

  • 每次数据更新前，会记录“前镜像”和“后镜像”。

  • 事务提交失败时，Seata 会自动回滚到前镜像。

🧪 应用案例：订单-库存-账户的事务控制

目标：

实现下单流程的全链路一致性（订单创建、库存扣减、账户扣款）

服务结构：

  • order-service：创建订单（TM）

  • stock-service：扣减库存（RM）

  • account-service：扣减余额（RM）

实现方式：

只需加一个 @GlobalTransactional 注解，Seata 自动识别、协调。

🛠️ Seata 配置关键要点

1. 配置 Seata Server（TC）：

  • 配置 registry.conf（注册中心，e.g., Nacos）

  • 配置 file.conf（事务日志、数据库等）

2. 各服务集成 Seata：

  • 引入 seata-spring-boot-starter

  • 使用 DataSourceProxy 代理数据源

  • 建表：undo_log

🧯 Seata 常见问题处理

问题 说明 解决建议
undo_log 表未建 无法记录回滚数据 每个 RM 数据库都要建
不支持的 SQL 如复杂 join 子查询 尽量用简单 SQL
网络抖动导致 RM 失败 事务挂起 配合服务降级+补偿机制
全局事务日志太多 性能影响 开启清理策略 + 持久化优化

总结：Seata 使用建议

建议 内容
使用模式 首选 AT 模式，轻量无侵入
数据源支持 常见的 MySQL、Oracle、PostgreSQL 支持良好
微服务调用 使用 Feign 或 RestTemplate 传递 XID
性能要求高 核心链路可用 TCC 精细控制
长流程业务 使用 Saga 模式编排状态和补偿
