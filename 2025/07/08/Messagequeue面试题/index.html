一、消息队列的作用与使用场景

优点

  • 解耦：系统间异步通信，降低服务间耦合度。

  • 削峰填谷：应对短时间内突发流量，缓冲系统压力。

  • 异步处理：提升响应速度，将耗时操作异步化。

  • 流量削峰：秒杀、电商抢购等高并发场景。

缺点

  • 系统复杂度上升（引入 MQ 中间件）

  • 消息可靠性、顺序性、重复消费等需额外处理

  • 消息堆积/丢失风险

  • 运维成本增加

常见使用场景

  • 用户注册发送短信邮件

  • 订单异步处理

  • 秒杀系统限流削峰

  • 日志收集系统（ELK）

  • 分布式事务协同（事务消息）

二、如何设计一个MQ（简版）

核心模块

  1. 消息模型设计：Topic / Queue / Delay / Dead Letter

  2. 存储机制：基于磁盘的日志存储（如 commitlog）

  3. Producer模块：支持同步、异步、批量发送、重试机制

  4. Consumer模块：支持广播/集群消费、幂等性、ACK确认机制

  5. Broker集群架构：支持高可用、分区、副本、负载均衡

  6. 事务支持（可选）：两阶段提交 / 半消息设计

  7. 可视化管理控制台：Topic管理、消费状态监控

三、消息可靠传输保障机制

生产者端保障

  • 消息持久化再发送（retry）

  • 发送确认机制（sync send）

  • 事务消息（半消息+二阶段提交）

Broker端保障

  • 消息持久化（落盘）

  • 副本同步（Kafka）

  • 主从同步与ACK确认（RocketMQ、Kafka）

消费者端保障

  • ACK确认消费

  • 重试机制与死信队列

  • 消费幂等性控制（去重）

四、如何保证不重复消费（幂等性）

  • 去重表/幂等表

  • 业务主键唯一约束（如订单号）

  • Redis SETNX、布隆过滤器等缓存控制

  • 消费者业务设计幂等

五、死信队列与延时队列

死信队列（DLQ）

产生场景：

  • 消息消费失败超过重试次数

  • 消息 TTL 到期

  • 队列满，无法投递

作用：

  • 异常消息归档、人工干预、告警分析

延时队列

用途：

  • 定时关闭订单

  • 延迟发送通知/重试机制

实现方式：

  • RabbitMQ：使用 TTL + 死信队列

  • RocketMQ：支持 18 个等级的延迟级别

  • Kafka：基于时间轮 / 自定义调度系统（如使用 delay topic+定时器）

六、RocketMQ 核心机制

持久化机制

  • 消息写入 commitlog（顺序追加）

  • 消息索引结构为 consumeQueue + indexFile

  • 同步/异步刷盘机制（flush）

架构设计

  • Producer：支持异步/同步/单向消息

  • Broker：Master/Slave 模式，支持 DLedger 主从同步

  • NameServer：注册中心 + 路由发现

  • Consumer：支持广播/集群模式，Pull模式消费

顺序消息实现

  • 将同一业务 key 的消息路由到同一 queue（分区）

  • 单线程串行消费该队列

RocketMQ事务消息原理

  • 半消息发送后，服务执行本地事务

  • 回调检查事务状态（回查机制）

  • 最终提交或回滚

七、RabbitMQ 核心机制

架构设计

  • Producer -> Exchange -> Queue

  • 支持主从（镜像队列）+ HA（Erlang 集群）

  • 使用 AMQP 协议

交换机类型

  • Direct：按 routing key 精准匹配

  • Fanout：广播给所有绑定队列

  • Topic：通配符匹配路由

  • Headers：按 headers 匹配

普通集群模式

  • 每台节点独立存储队列，非镜像

  • 消费只能从包含该队列的节点消费，非高可用

持久化机制

  • Exchange / Queue / Message 三者都设为 durable/persistent 才算真正持久化

  • 结合 confirm + ACK 控制消息投递

RabbitMQ 事务消息机制

  • 支持事务（txSelect/txCommit）但性能差

  • 推荐使用 Publisher Confirms（ConfirmCallback） + ACK机制替代

八、Kafka 核心机制

架构设计

  • Producer -> Broker(Partition) -> ConsumerGroup

  • Topic 分区、每个分区有副本（leader + follower）

  • 使用 pull模式 拉取数据（高吞吐）

副本同步机制

  • ISR（In-Sync Replicas）：leader + 跟上同步的 follower

  • ACK = all 时，写入所有 ISR 成员才确认成功

Rebalance机制

  • Group Coordinator 负责分配分区给消费者

  • 消费者上下线、topic 分区变化时触发 rebalance

  • 可选策略：Range / RoundRobin / Sticky

高性能原因

  • 磁盘顺序写 + PageCache

  • 零拷贝（SendFile）

  • 批量压缩 + 批量发送

  • Pull 拉模式 + Offset 控制由客户端维护

Kafka vs RocketMQ 吞吐对比

  • Kafka优势：磁盘 IO + 批量拉 + 分区多并发

  • RocketMQ：支持强事务、顺序消费、延迟队列更方便

九、MQ 的分布式事务与一致性保障

  • 方案一：本地消息表 + 定时任务补偿

  • 方案二：RocketMQ事务消息机制（二阶段）

  • 保证关键点：

    • 本地事务执行后发送 MQ 消息（确保原子性）

    • MQ 消息投递成功前不提交本地事务（或反之）

    • MQ 端有事务回查机制

十、产品选型建议（Kafka / RabbitMQ / RocketMQ）

特性 Kafka RabbitMQ RocketMQ
吞吐量 极高（适合大数据） 一般（10万级） 较高（适合高并发业务）
消息顺序 分区级顺序 支持，但较弱 队列级顺序控制
延迟队列 不支持（需自定义） 支持 TTL + DLX 支持 18 级别延迟
消息持久化 高性能、日志段机制 三重持久设置 CommitLog + ConsumeQueue
事务消息 无原生支持 有，但性能差 原生支持事务消息
易用性 学习成本略高 简单易用、社区活跃 阿里系生态，中文文档齐全
