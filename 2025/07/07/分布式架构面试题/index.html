🧱 一、Dubbo 核心原理与架构

1. Dubbo 是什么？能做什么？

Dubbo 是阿里开源的高性能 Java RPC 框架，支持服务注册发现、负载均衡、容错、服务治理等功能。它解耦了服务提供方与调用方，广泛用于微服务架构中。

2. Dubbo 的架构设计与分层

Dubbo 采用分层架构，主要包括：

层级 作用
接口层 定义业务接口
服务层 实现业务逻辑
配置层 配置服务暴露、消费方式
注册中心 服务注册与发现（如 ZooKeeper）
远程调用 序列化、网络传输、协议交互
监控中心 收集调用信息

3. Dubbo 支持的协议有哪些？

  • dubbo（默认）：基于 Netty，使用 Hessian2 序列化

  • rmi：JDK 原生远程调用

  • rest：基于 HTTP

  • http/json、thrift 等

4. Dubbo 支持的注册中心？

  • Zookeeper（最常用）

  • Nacos

  • Etcd

  • Consul

  • Redis（已废弃）

5. Dubbo 的负载均衡策略有哪些？

  • Random（默认）：按权重随机

  • RoundRobin：轮询，适合接口调用时间均衡的情况

  • LeastActive：最少活跃调用数

  • ConsistentHash：一致性哈希，适用于同参数请求路由到同一提供者

6. Dubbo 的集群容错策略有哪些？

  • Failover（默认）：失败自动重试（适合读操作）

  • Failfast：快速失败（适合写操作）

  • Failsafe：失败忽略

  • Failback：失败自动恢复（后台重试）

  • Forking：并行调用多个，只取一个成功

  • Broadcast：广播调用所有提供者

7. Dubbo 是如何完成服务导出的？

启动时通过 ServiceConfig.export() 暴露服务：

  • 构造 Invoker 对象（封装服务调用逻辑）

  • 通过 Protocol.export(invoker) 暴露为网络服务

  • 同时向注册中心注册服务

8. Dubbo 是如何完成服务引入的？

  • ReferenceConfig.get() 创建 Invoker，封装远程服务代理

  • Protocol.refer() 连接注册中心获取服务地址

  • 返回动态代理对象供消费者调用

9. Dubbo 支持 SPI 机制吗？为什么不用 JDK SPI？

Dubbo 实现了自己的 扩展机制（Dubbo SPI），比 JDK SPI 更强大：

  • 支持按条件激活扩展（@Adaptive）

  • 支持扩展点包装（Wrapper）

  • 支持自动注入依赖（Setter）
    JDK SPI 只能加载全量，且不支持懒加载与条件激活。

10. Dubbo 和 Spring Cloud 有什么区别？

特性 Dubbo Spring Cloud
服务调用 RPC（基于 Netty） REST（HTTP）
注册中心 Zookeeper、Nacos Eureka、Consul
协议 多协议支持（dubbo/rmi等） HTTP/JSON
扩展性 强，插件化、SPI机制 弱，依赖 Spring Cloud 生态
成熟度 企业内部流行（阿里、京东等） Spring 生态广泛，使用率高

🧭 二、Zookeeper 原理与应用

11. Zookeeper 的数据模型和节点类型？

ZK 是一棵树状结构，节点称为 ZNode，支持四种节点类型：

  • 持久节点（Persistent）

  • 临时节点（Ephemeral）

  • 顺序节点（Sequential）

  • 临时 + 顺序节点（Ephemeral Sequential）

12. Zookeeper 的典型应用场景？

  • 注册中心（Dubbo）

  • 分布式锁（临时有序节点）

  • 配置中心

  • 集群管理 / 主从选举

  • 分布式队列

13. Zookeeper 的 Watch 机制？

  • 客户端对节点注册监听

  • 节点变更（数据或子节点）触发事件

  • Watch 是一次性的，需手动重新注册

14. Zookeeper 的 ZAB 协议？

  • Zookeeper Atomic Broadcast，保证数据一致性

  • 类似 Raft 的协议，核心流程：

    • Leader 选举（先选 leader，再同步数据）

    • 提案广播（Proposal）

    • 半数写入即提交（Quorum）

15. Zookeeper 的选举过程？

  • 服务器启动后广播投票（myid + zxid）

  • 多轮投票直到半数选出 leader

  • 节点状态变为 LEADER、FOLLOWER、OBSERVER

16. Zookeeper 数据同步原理？

  • Leader 接收写请求，生成事务提案 Proposal

  • 广播 Proposal 给 follower

  • 半数 follower Ack 后，Leader commit 并通知 follower 提交

17. zk 中一个客户端修改节点数据，其他客户端能马上看到吗？

是的，但依赖 Watch + 通知机制，不是“主动推送”，而是注册后才会感知变更。

18. Zookeeper 如何实现分布式锁？

  • 利用“临时有序节点” + 最小节点监听机制

  • 所有客户端创建 /lock/lock-xxxx 节点

  • 获取最小序号即获得锁

  • 节点断开连接自动删除

19. ZK 如何存储 Dubbo 的注册信息？

  • /dubbo/com.xxx.Service/providers：提供者 URL

  • /dubbo/com.xxx.Service/consumers：消费者 URL

  • /dubbo/com.xxx.Service/configurators：配置项

20. Zookeeper 和 Eureka 区别？

特性 Zookeeper Eureka
角色 通用协调服务 专门的服务注册发现中心
CAP CP AP（自我保护机制）
Watch 有 无
健康检查 被动 主动

🔁 三、分布式事务 / 锁 / ID / 会话

21. 分布式事务有哪些实现方案？

  • 2PC（两阶段提交） → 同步阻塞

  • 3PC（三阶段） → 改进了阻塞问题

  • TCC（Try-Confirm-Cancel） → 补偿机制

  • 消息最终一致性：本地消息表 / 可靠消息投递

  • Saga 模式：长事务链式补偿

22. 简述 TCC 模型？

TCC 将事务拆分成三个阶段：

  • Try：预留资源

  • Confirm：正式提交

  • Cancel：回滚释放资源

适用于资源可预留、操作幂等的场景。

23. 分布式锁实现方式？

  • 基于 Zookeeper（临时有序节点）

  • 基于 Redis（setnx + lua 脚本 + UUID）

  • 基于数据库（悲观/乐观锁）

24. Redis 分布式锁有哪些注意点？

  • 加锁：SET key value NX PX

  • 解锁：Lua 脚本校验 value 防止误删

  • 防止死锁：设置超时时间 + watchdog

25. 分布式 ID 方案有哪些？

  • UUID：无序、不可读

  • 数据库自增：易瓶颈

  • Redis + INCR：简单但易失效

  • 雪花算法（Snowflake）：64位整型，包含时间戳、机器号、序列号

26. 雪花算法原理？

  • 41bit 时间戳 + 10bit 机器ID + 12bit 自增序号 + 1bit 符号位

  • 单机每毫秒可生成 4096 个唯一 ID

27. 接口幂等性如何实现？

  • 幂等唯一 token（防止重复提交）

  • 数据库唯一索引约束

  • 状态机控制（重复请求检查状态）

  • 乐观锁机制（version 字段）

28. 分布式系统下 session 共享的解决方案？

  • Session Sticky：请求打到同一个节点（不推荐）

  • 共享存储：如 Redis 存储 Session

  • JWT + Token：前后端分离场景使用多

🧮 四、负载均衡与一致性

29. 常见的负载均衡算法？

  • 随机（Random）

  • 轮询（Round Robin）

  • 最小连接数（Least Connection）

  • 哈希一致性（Consistent Hash）

  • 加权随机 / 加权轮询

30. 分布式缓存寻址算法有哪些？

  • 哈希取模：简单但不支持节点动态变化

  • 一致性哈希：节点变化只影响部分 key

  • 虚拟节点：提升负载均衡性与稳定性

🧠 五、CAP / BASE / 一致性协议 / RPC

31. 什么是 CAP 理论？

在分布式系统中，最多只能同时满足以下三个中的两个：

  • Consistency（一致性）

  • Availability（可用性）

  • Partition Tolerance（分区容错性）

32. 什么是 BASE 理论？

对 CAP 的弱化：

  • Basically Available（基本可用）

  • Soft-state（软状态）

  • Eventual Consistency（最终一致）

常用于大规模分布式系统，如电商、缓存等。

33. 简述 Paxos、Raft、ZAB 协议？

  • Paxos：理论完整，复杂度高

  • Raft：实现简单，分为 Leader 选举、日志复制、状态机同步

  • ZAB：Zookeeper 的协议，改进型 Raft，支持崩溃恢复与观察者

34. 简述 RPC、RMI 是什么？

  • RPC（远程过程调用）：像调用本地函数一样调用远程方法

  • RMI：Java 原生 RPC 方案，基于对象传输，性能低于 RPC 框架

🧵 六、其他补充题

35. ZK集群挂了还能通信吗？

  • 如果 ZK 完全不可用，Dubbo 无法完成服务注册与发现

  • 已建立连接的服务间可继续通信（缓存 URL）

  • 重启或服务变动时会失败

36. 为什么 Zookeeper 可用于注册中心？

  • 数据一致性强（ZAB）

  • Watch 机制通知变更

  • 支持临时节点自动失效

  • 结构简单，天生适合服务目录管理
